## 1. 두 스킬 그룹의 호출 방식 차이

설치되는 10개 스킬은 두 출처에서 옵니다. **호출 방식이 다릅니다.**

| 그룹 | 출처 | 호출 방식 | 예시 |
|------|------|----------|------|
| 의사결정 (4종) | gstack | 슬래시 명령 | `/office-hours` |
| 실행 (4-6종) | Superpowers | 자연어 트리거 또는 명시적 호출 | "테스트 먼저 작성해줘" → `test-driven-development` 자동 트리거 |

**의사결정 스킬**: 슬래시 명령을 직접 타이핑해야 활성화됩니다.

**실행 스킬**: Claude가 작업 문맥을 보고 자동으로 트리거합니다. 명시적으로 부르려면 "writing-plans 스킬을 적용해서 분해해줘"처럼 스킬명을 언급합니다.

## 2. 작업 종류별 권장 워크플로우

작업 크기·성격에 따라 다른 진입점을 사용합니다.

### 2.1 명확한 버그 즉시 수정 (1-3줄 수정)

```
사용자: "이 함수의 null check 빠진 거 고쳐줘"
   ↓
Claude가 자연어로 직접 수정
   ↓ (자동 동반)
test-driven-development        # 테스트 먼저
verification-before-completion # 빌드/테스트 출력 확인
```

`/office-hours`, `/plan-*`은 불필요. 의식이 과합니다.

### 2.2 작은 UX 개선 (버튼 색, 마진, 라벨 등)

```
사용자: "헤더 폰트 크기 조금 키워줘"
   ↓
Claude가 자연어로 직접 수정
   ↓
verification-before-completion # 시각 확인
```

### 2.3 기존 기능 리팩토링

```
"이 클래스를 두 개로 쪼개고 싶어"
   ↓
writing-plans                  # 한입 크기 태스크 분해
   ↓
executing-plans                # 분해된 태스크 순차 실행
   ↓
test-driven-development        # 태스크별 RED-GREEN-REFACTOR
   ↓
verification-before-completion # 전체 통과 확인
```

### 2.4 새 기능 추가 (작음)

```
/plan-eng-review               # 아키텍처·테스트 매트릭스 잠금
   ↓
writing-plans
   ↓
executing-plans + test-driven-development
   ↓
verification-before-completion
```

### 2.5 새 기능 추가 (큼) — 전체 7단계

```
/office-hours        # 진짜 만들 것 재정의 (6 forcing question)
   ↓
/plan-ceo-review     # 10-star 제품 비전 결정
   ↓
/plan-eng-review     # 기술 설계 잠금
   ↓
writing-plans        # 태스크 분해
   ↓
executing-plans + test-driven-development
   ↓
verification-before-completion
```

각 단계 후 사용자 승인을 기다립니다. 자동 직진 안 됨.

### 2.6 원인 불명 버그 추적

**기본 셋업**:

```
/plan-eng-review               # 원인 가설 후보 잠금
   ↓
writing-plans → executing-plans → TDD → verification
```

**`--with-debug` 옵션 켜진 경우**:

```
systematic-debugging           # 4단계 root-cause (자동 우선 트리거)
   ↓
이후 흐름 동일
```

### 2.7 성능 최적화

```
/plan-eng-review               # 측정 기준·타겟 명시
   ↓
test-driven-development        # 벤치마크 테스트 먼저
   ↓
executing-plans                # 최적화 단계별 실행
   ↓
verification-before-completion # 측정값 비교
```

## 3. 옵션 플래그가 켜졌을 때 흐름 변화

### 3.1 `--with-debug`

`systematic-debugging` 스킬이 추가됩니다. 원인 불명 버그를 만나면 `/plan-eng-review` 대신 이게 먼저 활성화되어 4단계(증상 격리 → 가설 → 검증 → 수정)를 거친 후 평소 흐름에 합류합니다.

### 3.2 `--with-review`

모든 흐름의 `verification-before-completion` 이후에 코드 리뷰 게이트가 추가됩니다.

- **`/review`** (gstack): PR 머지 전 게이트로 사용. 슬래시 명령으로 직접 호출.
- **`requesting-code-review`** (Superpowers): TDD 흐름과 통합된 리뷰 요청. 자연어 트리거.

PR을 만들기 전이나 큰 변경 commit 전에 권장합니다.

### 3.3 `--with-all`

위 두 옵션 모두 적용. 본인 연구실용 풀세트입니다.

## 4. 실전 시나리오

### 시나리오 A: 큰 신규 기능 ("사용자 권한 관리 시스템 추가")

```
1. /office-hours
   → "권한 관리"가 진짜 풀어야 할 문제인지 재정의
   → 6 forcing question (demand reality, narrowest wedge 등)
   → 산출: 무엇을 만들 것인지 1문장 정의

2. /plan-ceo-review
   → 스코프 결정 (RBAC vs ABAC, OAuth 통합 여부 등)
   → 4가지 모드 중 선택 (HOLD SCOPE 추천)
   → 산출: 비전 1쪽

3. /plan-eng-review
   → 아키텍처 (테이블 스키마, API 엔드포인트, 미들웨어)
   → 엣지 케이스, 테스트 매트릭스
   → 산출: 기술 설계 문서

4. "writing-plans 스킬로 위 설계를 한입 크기 태스크로 쪼개줘"
   → 2-5분 단위 태스크 리스트 (DRY/YAGNI 강제)
   → 산출: tasks.md (체크리스트)

5. "executing-plans로 1번 태스크부터 진행해줘"
   → 각 태스크 시작 시 TDD 흐름으로 자동 진입
   → RED → GREEN → REFACTOR

6. 모든 태스크 완료 시 verification-before-completion 자동 트리거
   → 빌드, 테스트, 린트 출력 확인 후 완료 선언
```

### 시나리오 B: 명확한 버그 ("로그인 폼 enter 키 동작 안 함")

```
사용자: "로그인 폼에서 enter 누르면 submit 되도록 고쳐줘"
   ↓
Claude: <form> 태그 누락 확인 → 수정
        test-driven-development 자동 트리거 → 키보드 이벤트 테스트 추가
        verification-before-completion 자동 트리거 → 테스트 통과 확인
   ↓
완료. /office-hours, /plan-* 거치지 않음.
```

### 시나리오 C: 디버깅 (`--with-debug` 켜짐, "어제까지 되던 기능이 안 됨")

```
사용자: "/users API 호출이 갑자기 500 에러를 뱉어"
   ↓
systematic-debugging 자동 트리거
   1단계 증상 격리: 어느 환경에서? 어느 입력에서? 언제부터?
   2단계 가설 수립: 최근 변경 사항, 가능한 원인 3-5개
   3단계 검증: 가설별로 재현·반증 시도
   4단계 수정: 근본 원인 확정 후 패치
   ↓
test-driven-development: 회귀 테스트 추가
   ↓
verification-before-completion: 통과 확인
```

## 5. 자주 묻는 질문

### Q. 작은 작업에서 `/plan-*` 같은 무거운 스킬이 자꾸 활성화되면?

명시적으로 우회하면 됩니다.

```
"이건 한 줄 수정이니까 brainstorming 건너뛰어"
"그냥 고쳐줘"
"plan-eng-review 없이 바로 수정"
```

결과 지향 표현일수록 무거운 의사결정 스킬이 덜 트리거됩니다.

### Q. Superpowers 스킬을 명시적으로 부르고 싶으면?

스킬명을 직접 언급하세요.

```
"writing-plans 스킬을 적용해서 이 작업을 분해해줘"
"test-driven-development으로 진행해줘"
"verification-before-completion을 돌려서 완료 확인해"
```

### Q. 의존성 점검에서 미설치 명령 경고가 나오는데 문제 있나?

정상입니다. gstack 의사결정 스킬은 본문에서 다음 단계로 `/qa`, `/ship` 같은 명령을 추천하는데, 본 인스톨러는 이들을 의도적으로 제외했습니다. 추천이 자동 이어지지 않을 뿐 스킬 자체 동작은 정상입니다.

### Q. `CLAUDE.md`에 스킬 목록을 적어야 하나?

권장합니다. Claude가 더 안정적으로 스킬을 인식합니다.

- **기본 7개 설치**: [README.md](README.md)의 "설치 후 권장 작업" 블록을 복사
- **`--with-*` 옵션 설치**: 인스톨러가 설치 마지막에 출력하는 블록을 복사 (옵션에 따라 자동 갱신됨)

### Q. 한 스킬을 빼고 싶으면?

`install-dev-skills.sh` 상단의 `GSTACK_SKILLS` 또는 `SUPERPOWERS_SKILLS` 배열에서 해당 줄을 제거한 후 `--remove`로 정리하고 다시 설치합니다.

### Q. 풀세트 gstack/Superpowers와 같이 설치해도 되나?

가능합니다. 단, 같은 이름의 스킬이 두 곳에 있으면 `~/.claude/skills/<name>/`이 우선합니다(본 인스톨러가 만드는 위치). `--update`로 본 인스톨러 셋업을 덮어쓸 수 있습니다.

### Q. 어떤 작업에 어떤 스킬을 써야 할지 헷갈리면?

이 문서의 "2. 작업 종류별 권장 워크플로우"를 보고 가장 가까운 케이스로 매핑하세요. 매핑이 애매하면 그냥 자연어로 작업을 설명하면 Claude가 적절한 스킬을 트리거합니다.
